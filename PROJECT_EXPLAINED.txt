1. PROJECT OVERVIEW
================================================================================

This is an AI-powered Customer Support Agent that can handle real customer
service tasks autonomously — checking order statuses, processing refunds,
retrieving company policies, escalating to human managers when needed, and
blocking malicious inputs. Think of it as a smart chatbot for an e-commerce
company that knows the company's policies and can take actions on behalf
of the customer.

The key innovation is "Human-in-the-Loop" governance: the agent can handle
routine requests (like small refunds under $10) fully automatically, but
for high-risk actions (like large refunds or legal threats), it pauses and
asks a human manager to approve or reject the action before proceeding.

2. HOW IT WORKS - ARCHITECTURE & FLOW
================================================================================

The system has three main layers:

  FRONTEND (Streamlit UI)
         |
         v
  BACKEND (FastAPI API)
         |
         v
  AGENT CORE (LangGraph State Machine)

--- Agent Processing Pipeline ---

When a customer sends a message, it flows through these stages:

  STAGE 1: PREPROCESS (Risk Scanning)
    - Scans the message for risk keywords: "sue", "lawyer", "legal",
      "refund", "compensation"
    - If it finds legal keywords OR the refund amount exceeds $10, it
      flags requires_human_approval = True
    - Extracts order ID (like A4821) and refund amount ($5, $300, etc.)
    - Uses LLM (GPT-4o-mini) for structured risk analysis if available,
      falls back to keyword matching if LLM is unavailable

  STAGE 2: PLANNER (Intent Recognition)
    - Determines what the customer wants:
        * "order_status" - they want to check an order
        * "refund" - they want money back
        * "general" - general question
        * "out_of_scope" - something we can't help with (competitor comparison)
    - Retrieves relevant company policies from ChromaDB vector database
    - Plans the next action: order_lookup, policy_check, refund, or refuse

  STAGE 3: HUMAN GATE (Approval Check) [conditional]
    - Only activated when requires_human_approval = True
    - PAUSES the entire workflow using LangGraph's interrupt() mechanism
    - Waits for a human manager to either APPROVE or REJECT
    - If approved: continues to tool execution
    - If rejected: ends with a polite refusal message

  STAGE 4: TOOL EXECUTOR (Action Execution)
    - Runs the actual tools based on the planned action:
        * check_order_status() - looks up order in database
        * apply_refund() - processes refund (only if approved or under $10)
        * send_goodwill_credit() - issues store credit for late orders
        * refuse_out_of_scope() - politely declines off-topic requests
    - After tool execution, runs COMPLIANCE CHECK:
        * PII scan (ensures no credit card numbers or other customers' data)
        * Authorization check (ensures refund promises match approval status)

  STAGE 5: RESPONSE
    - The agent's response is sent back through the pipeline
    - Every step is logged to the audit trail

3. TECH STACK & WHY EACH TECHNOLOGY WAS CHOSEN
================================================================================

--- Core Agent Framework ---

  LangGraph (>=0.2.0)
    WHY: Chosen because it provides a state-machine approach to building
    AI agents. Unlike simple chain-based agents, LangGraph allows:
    - Conditional routing (different paths based on risk level)
    - Human-in-the-loop interrupts (pausing for approval)
    - State persistence across sessions (MemorySaver checkpointer)
    - Clear visualization of agent workflow
    This is critical for a support agent that needs governance controls.

  LangChain (>=0.3.0)
    WHY: Provides the foundational abstractions for:
    - Message handling (HumanMessage, AIMessage)
    - Tool definitions (@tool decorator)
    - LLM integration (ChatOpenAI)
    - Document handling for RAG
    LangGraph builds on top of LangChain, so they work together naturally.

  LangChain-OpenAI (>=0.2.0)
    WHY: OpenAI connector for LangChain. Used for:
    - GPT-4o-mini for intent recognition and risk analysis
    - Structured output parsing (AnalysisPlan, RiskScan models)
    - Embedding generation for vector search

--- API & Frontend ---

  FastAPI (>=0.115.0)
    WHY: High-performance Python web framework for the REST API.
    - Provides /chat, /approve, /get_audit_log, /health endpoints
    - Automatic OpenAPI documentation
    - Pydantic validation for request/response models
    - CORS middleware for cross-origin requests

  Uvicorn (>=0.32.0)
    WHY: ASGI server to run FastAPI. Fast, production-ready.

  Streamlit (>=1.40.0)
    WHY: Rapid UI prototyping framework for the dashboard.
    - Single-file Python UI (ui.py) — no HTML/CSS/JS needed
    - Session state management for multi-turn conversations
    - Built-in widgets (buttons, text inputs, tabs, spinners)
    - Custom CSS injection for the futuristic "AI Command Center" look
    - Real-time workflow visualization with SVG path nodes

--- Data & Validation ---

  Pydantic (>=2.9.0)
    WHY: Data validation and serialization.
    - AgentState TypedDict for LangGraph state schema
    - FastAPI request/response models (ChatRequest, ChatResponse)
    - Structured output models (AnalysisPlan, RiskScan)
    - Ensures type safety across the entire system

  python-dotenv (>=1.0.0)
    WHY: Loads environment variables from .env file.
    - API keys (OPENAI_API_KEY, GOOGLE_API_KEY, GITHUB_TOKEN)
    - Configuration (CHROMA_PERSIST_DIR, EMBEDDING_MODEL)
    - Keeps secrets out of code

--- RAG / Vector Store ---

  ChromaDB (>=0.5.0)
    WHY: Lightweight vector database for policy document retrieval.
    - Stores embedded policy documents (refund, shipping, privacy)
    - Enables similarity search for relevant policies
    - Persistent storage (./data/chroma_db/)
    - Self-healing: auto-recreates if database is corrupted

  LangChain-Chroma (>=0.1.0)
    WHY: LangChain integration for ChromaDB.
    - Wraps ChromaDB with LangChain's vectorstore interface
    - Handles embedding function management

  LangChain-Google-Genai (>=2.0.0)
    WHY: Google Gemini embeddings (free tier: 1500 req/day).
    - Primary embedding provider (gemini-embedding-001)
    - Falls back to OpenAI, GitHub Models, or local HashingEmbeddings

--- Testing ---

  Pytest (>=8.3.0)
    WHY: Python testing framework.
    - Unit tests for tools, RAG, governance
    - Integration tests for the full agent pipeline
    - Evaluation suite (eval_suite.py) with 5 capstone scenarios

  httpx (>=0.27.0)
    WHY: Async HTTP client for testing FastAPI endpoints.

  pytest-asyncio (>=0.24.0)
    WHY: Async test support for testing async endpoints.

4. MODELS & AI COMPONENTS USED

--- LLM Models ---

  GPT-4o-mini (OpenAI / GitHub Models)
    PURPOSE: Agent reasoning and structured output
    - Risk analysis (structured RiskScan output)
    - Intent classification (structured AnalysisPlan output)
    - Fallback: regex/keyword matching if LLM unavailable
    - Temperature: 0.0 (deterministic)
    - Max retries: 0 (fail fast)

  --- Embedding Models ---

  Google Gemini gemini-embedding-001 (Primary)
    - 3072-dimensional embeddings
    - Free tier: 1500 requests/day
    - Used for policy document vectorization

  OpenAI text-embedding-3-small (Fallback)
    - Standard OpenAI embeddings
    - Used when Gemini is unavailable

  GitHub Models text-embedding-3-small (Fallback)
    - Free via GitHub PAT
    - Used when OpenAI key unavailable

  Local HashingEmbeddings (Offline Fallback)
    - 128-dimensional feature hashing
    - Adler32 hash for word-level embeddings
    - L2 normalized
    - Used when ALL API providers are unreachable
    - WARNING: Not compatible with API-generated embeddings

--- Structured Output Models (Pydantic) ---

  RiskScan:
    - legal_threat: bool
    - detected_keywords: list[str]
    - refund_requested: bool

  AnalysisPlan:
    - intent: str (order_status/refund/general/out_of_scope)
    - order_id: str
    - refund_amount: float

5. KEY ELEMENTS & CONCEPTS CONSIDERED
================================================================================

--- Governance & Safety ---

  1. HUMAN-IN-THE-LOOP (HITL)
     - High-value refunds (>$10) require human approval
     - Legal threats automatically escalate
     - Uses LangGraph's interrupt() for pausing/resuming
     - Manager can approve or reject via UI buttons

  2. RISK SCANNING
     - Keyword-based detection: sue, lawyer, legal, refund, compensation
     - LLM-enhanced structured analysis when available
     - Risk levels: LOW, MEDIUM, HIGH, CRITICAL
     - Automatic escalation for CRITICAL risk

  3. COMPLIANCE & PII PROTECTION
     - Scans responses for credit card numbers (13-16 digits)
     - Blocks disclosure of other customers' order IDs
     - Prevents unauthorized refund promises
     - Modified responses replace non-compliant content

  4. AUDIT TRAIL
     - Every step logged to data/audit_log.json
     - Timestamps, risk levels, actions, details
     - Viewable in UI's Audit Log tab
     - Retained for compliance and debugging

--- RAG (Retrieval-Augmented Generation) ---

  1. POLICY RETRIEVAL
     - Company policies stored in ChromaDB vector database
     - Similarity search finds relevant policy snippets
     - Policies: refund_policy.md, shipping_policy.md, privacy_policy.md
     - Self-healing: auto-recreates index if empty

  2. INGESTION PIPELINE
     - Loads markdown files from data/policies/
     - Loads flat text from data/policies.txt
     - Chunks using MarkdownTextSplitter (300 chars, 30 overlap)
     - Embeds and stores in ChromaDB

  3. EMBEDDING FALLBACK CHAIN
     Priority 1: Google Gemini (free tier)
     Priority 2: OpenAI/OpenRouter API
     Priority 3: GitHub Models (free via PAT)
     Priority 4: OpenAI direct
     Fallback: Local HashingEmbeddings (128-dim)

--- State Management ---

  AgentState (TypedDict):
    - messages: list[BaseMessage] — conversation history
    - order_id: str — extracted order ID
    - refund_amount: float — extracted refund amount
    - requires_human_approval: bool — governance flag
    - audit_log: list[dict] — step-by-step audit trail
    - gate_response: dict — human gate status
    - approval_status: str — approved/rejected/pending
    - detected_risk_keywords: list[str] — risk scan results

8. KEY DESIGN DECISIONS
================================================================================

  1. WHY LangGraph over simple LangChain chains?
     -> Need conditional routing, human-in-the-loop, and state persistence.

  2. WHY $10 auto-refund threshold?
     -> Balances customer satisfaction (fast small refunds) with financial
        safety (large refunds need human oversight).

  3. WHY ChromaDB over Pinecone/Weaviate?
     -> Lightweight, local-first, no cloud dependency. Perfect for a
        capstone project and demo.

  4. WHY fallback chain for embeddings?
     -> Ensures the system works even without API keys (offline mode).
        Different users may have different API access levels.

  5. WHY both FastAPI AND Streamlit?
     -> FastAPI provides a clean REST API for programmatic access.
        Streamlit provides the visual demo dashboard. They complement
        each other for different use cases.

  6. WHY mock data instead of a real database?
     -> Simplifies setup and demo. No database server needed.
        Easy to understand and modify for testing.

  7. WHY structured output (Pydantic models) for LLM?
     -> Ensures the LLM returns consistent, parseable responses.
        Reduces errors from free-form text generation.

  8. WHY audit trail at every step?
     -> Enterprise requirement. Enables debugging, compliance auditing,
        and demonstration of agent decision-making process.

  9. WHY Playwright E2E tests?
     -> Verifies the full user experience in a real browser.
        Catches UI rendering issues that unit tests miss.
        Streamlit's shadow DOM requires special handling via
        [data-testid="stMarkdownContainer"] locators.
